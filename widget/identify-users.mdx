Identifying users lets you tie every feedback submission, vote, and comment to a real account in your system. You get accurate attribution in the dashboard, and users see their own history when they open the widget.

---

## Identify logged-in users

Call `identify` after login and any time the user's session changes.

```js
Quackback("identify", {
  id: "user_123",
  email: "ada@example.com",
  name: "Ada Lovelace",
});
```

| Field   | Type   | Required | Notes                                  |
|---------|--------|----------|----------------------------------------|
| `id`    | string | Yes      | Stable, unique ID from your system     |
| `email` | string | Yes      | Used for notifications and dedup       |
| `name`  | string | No       | Display name shown in the dashboard    |

<Note>
If you have **Verified Identity** enabled (recommended for production), skip this unverified call and use `ssoToken` instead — see the section below.
</Note>

---

## Identify anonymous users

Pass `{ anonymous: true }` for users who aren't logged in. The widget assigns them a temporary ID so their session is still tracked.

```js
Quackback("identify", { anonymous: true });
```

Anonymous activity merges into the user's account automatically when they log in and you call `identify` with their real ID.

<Tip>
A common pattern is to identify logged-in users with their details and fall back to anonymous for everyone else. That way the widget is always visible, but logged-in users get their feedback attributed.
</Tip>

---

## Clear identity on logout

Pass `null` to reset the widget to an unidentified state. Call this in your logout handler.

```js
Quackback("identify", null);
```

If the widget panel is open, it closes automatically before the trigger hides.

## Verified identity with JWT

Unverified `identify` calls trust the client. A determined user could call `Quackback("identify", { id: "someone-elses-id", ... })` to spoof another account. Verified identity prevents this by requiring a server-signed JWT.

Enable it in **Admin > Settings > Widget**, then copy the **Widget Secret**.

<Warning>
Never expose the widget secret in client-side code. Sign tokens on your server only.
</Warning>

### How it works

Your server signs a JWT with HS256 using the widget secret. The widget sends `ssoToken` to Quackback, which verifies the signature before accepting the identity claims.

### JWT claims

| Claim       | Required    | Notes                                       |
|-------------|-------------|---------------------------------------------|
| `sub`       | Yes         | User's stable ID                            |
| `email`     | Yes         | User's email address                        |
| `name`      | No          | Display name                                |
| `avatarUrl` | No          | URL of the user's avatar image              |
| `exp`       | Recommended | Unix timestamp; use ~5 minutes from now     |
| custom      | No          | Any additional key-value pairs (see below)  |

### Server-side signing examples

<Tabs>
<Tab title="Next.js">
```ts
// app/api/widget-token/route.ts
import { SignJWT } from "jose";
import { getServerSession } from "next-auth";

const secret = new TextEncoder().encode(process.env.QUACKBACK_WIDGET_SECRET);

export async function GET() {
  const session = await getServerSession();
  if (!session?.user) {
    return new Response("Unauthorized", { status: 401 });
  }

  const token = await new SignJWT({
    sub: session.user.id,
    email: session.user.email,
    name: session.user.name,
  })
    .setProtectedHeader({ alg: "HS256" })
    .setExpirationTime("5m")
    .sign(secret);

  return Response.json({ token });
}
```

def widget_token(request):
    payload = {

### Pass the token to the widget

Fetch the token from your endpoint and pass it as `ssoToken`.

```js
const { token } = await fetch("/widget-token").then((r) => r.json());

Quackback("identify", { ssoToken: token });
```

<Tip>
Fetch a fresh token on each login or page load. Tokens expire quickly by design — this limits the blast radius if one is intercepted.
</Tip>

## Custom attributes in the JWT

Add any extra claims to the JWT payload to attach custom attributes to the user. Attributes must be configured in **Admin > Settings > User Attributes** before they appear in the dashboard.

```js
// Server-side — add custom claims alongside the standard ones
const payload = {
  sub: user.id,
  email: user.email,
  name: user.name,
  // Custom attributes
  plan: "pro",
  company: "Acme Corp",
  mrr: 299,
};
```

Custom claims are validated against your configured user attributes. Claims that don't match a configured attribute are ignored, not rejected.

<Note>
Set up your attributes in **Admin > Settings > User Attributes** first so nothing gets silently dropped.
</Note>

## Rotate the secret

If your secret is ever exposed, rotate it immediately:

1. Go to **Admin > Settings > Widget** and click **Regenerate Secret**.
2. Deploy updated server-side token signing with the new secret.
3. The old secret is invalidated instantly — users will see an auth error until your server is updated.

<Warning>
Rotate and deploy atomically. Any lag between invalidating the old secret and deploying the new signing code will cause identify calls to fail for active users.
</Warning>
